# 鐵道尋蹤系統 — 功能與流程優化建議

> 本文件聚焦於系統本身的功能完整度、使用者體驗與流程效率，從旅客端、站務端、資料追蹤、配送機制、資料治理五個層面整理優化方向，並標註優先級供後續開發排序參考。

---

## 一、旅客端：從「被動搜尋」到「主動協尋」

### 1. 搜尋體驗：漸進式問答取代靜態篩選器

**問題**：目前搜尋邏輯（分類、車次、日期區間 + 以圖找圖）假設旅客記得精確資訊，但實際上旅客剛丟東西時記得的通常是模糊片段（大概路段、東西外觀、顏色）。

**優化方向**：
- 不要一開始就給旅客一排篩選欄位，改成對話式收斂流程：
  1. 你是要找東西，還是看到別人遺失的東西？
  2. 大概什麼時候、什麼路線/車站？（允許模糊範圍）
  3. 用文字描述或上傳照片
- 系統先用前面步驟縮小資料庫範圍，再做 AI 比對，準確率會比「在全資料庫中比對單張照片」更高
- 這是現有功能的**重組**而非新技術導入，改動成本相對低

### 2. 許願池：補上「即將失效」提醒機制

**問題**：目前邏輯是新物品入庫時比對許願池並發信，但缺少「協尋進度」與「期限提醒」，旅客登記後不知道該繼續等還是已經沒希望。

**優化方向**：
- 許願池登記後顯示「目前資料庫中相似物品數」（即使未完全比對成功，提供粗略提示）
- 加入「協尋期限」概念，到期前主動提醒旅客是否要延長協尋

### 3. 以圖找圖：建立信任與回饋機制

**問題**：AI 比對結果直接列出一排物品，若包含不相關項目，旅客會質疑系統準確度。

**優化方向**：
- 比對結果附上相似度分數並排序
- 低於信心門檻的結果歸入「可能相關（不太確定）」分組，避免旅客誤解系統判斷
- 讓旅客可對結果按「這不是我的東西」負反饋，此資料可用於未來校正比對邏輯（目前系統完全缺少使用者回饋迴圈）

## 二、站務端：把「登錄」變成「半自動化流程」

### 1. AI 輔助登錄：建立人工修正回饋機制

**問題**：Gemini 自動產生分類/標籤/描述後，若站務人員修正，這個修正結果目前未被系統記錄利用，等於浪費每一次校正訊號。

**優化方向**：
- 記錄「AI 原始判斷」vs「人工修正結果」的差異
- 長期可用於分析 AI 容易出錯的類別（例如顏色判斷、材質誤判），逐步優化辨識邏輯或 prompt 設計
- 低成本、高長期價值的資料累積動作

### 2. 危險品/違禁品過濾：改為分級攔截

**問題**：目前僅以關鍵字過濾擋掉登錄，遇到模糊個案（打火機、瑞士刀等）站務人員無從判斷如何處理。

**優化方向**：分為三級處理
- **禁止登錄**：明確違禁品（毒品、武器等）
- **需上報主管**：模糊地帶（刀具、易燃物），允許登錄但標記待審核
- **正常登錄但加警示標籤**：附帶處理 SOP 提示文字（例如「偵測為刀具類物品，請依規定通報○○單位」）

讓系統從單純資料庫升級為流程指引工具。

### 3. 易腐壞食物：建立獨立短期保管機制

**問題**：目前僅自動標記，缺少後續處理流程（保管期限、優先通知機制）。

**優化方向**：
- 易腐壞食物應有獨立的「短期保管倒數」，遠短於一般物品期限（例如 24-48 小時）
- 介面上以視覺提示凸顯「需優先處理」，避免被淹沒在一般物品清單中
- 若有對應車次/座位資訊，可評估是否能主動通知（需考量實際聯絡方式取得的可行性）

### 4. 批次操作安全網

**問題**：批次刪除/批次更新狀態提升效率，但批次刪除是高風險操作，缺乏防呆機制。

**優化方向**：
- 批次刪除前要求二次確認，並列出將被刪除的物品摘要清單
- 考慮採用軟刪除（soft delete：標記為已刪除但保留資料一段時間），避免誤刪無法復原

## 三、QR Code 流向追蹤：由「靜態紀錄」轉為「結構化軌跡」

### 1. 建立轉移紀錄資料表（ItemTransferLog）

**問題**：目前掃描後僅覆寫物品「目前所在車站」狀態，無法重建完整移動軌跡；若某環節未掃描，資訊會直接斷掉。

**優化方向**：
- 新增 `ItemTransferLog`（或類似命名）資料表，每次掃描新增一筆紀錄（時間、車站、操作人員），而非覆寫單一欄位
- 效益：
  - 可重建完整物品移動軌跡，方便追蹤異常環節
  - 可用於分析平均流轉時間，找出流程瓶頸站點

### 2. 狀態轉換合理性檢查

**問題**：配送狀態流程為 pending → delivering → arrived，若掃描順序錯亂（如未經 delivering 直接標記 arrived），目前可能缺乏防呆。

**優化方向**：
- 加入狀態機驗證邏輯，不合理的狀態跳轉應被攔截並提示

## 四、跨站配送：規則由「硬編碼」改為「可配置化」

**問題**：目前限制規則（例如 3C 電子產品不受理跨站配送）疑似寫死在程式邏輯中，未來規則調整需改動程式碼。

**優化方向**：
- 將配送限制規則抽出為設定項（例如規則表：分類 + 是否允許配送 + 限制原因），方便未來調整
- 補強敏感物品（證件）配送的領取驗證流程：旅客領取時應輸入驗證碼或其他身分確認機制，而非任何人可直接領走（目前疑似缺少此環節，屬於個資安全缺口）

## 五、資料治理與系統健康度

### 1. 敏感物品存取控制

**問題**：敏感物品目前僅做「標記」，未明確定義誰能查看完整證件/卡號資訊；若所有登入站務人員皆可看到完整末四碼，屬於個資風險。

**優化方向**：
- 敏感物品的完整資訊存取應有紀錄（誰、何時查看過）
- 考慮分權限設計：一般站務人員僅能看到「已標記為敏感物品」，需更高權限才能查看具體號碼內容

### 2. 登入機制強化

**問題**：目前為單一帳號密碼（`admin`/`admin123`）+ Cookie 驗證，無法追溯具體操作人員，也無法依職責分權，對多人協作場景風險較高。

**優化方向（依重要性排序）**：
- 至少建立「多帳號 + 操作紀錄」，可追溯哪個站務人員執行了新增/刪除/更新等操作
- 中期評估角色分級（一般站務 vs 主管），例如批次刪除、敏感物品檢視需主管權限

### 3. 資料庫索引與查詢效能

**問題**：隨資料量增加，常用篩選欄位若無索引，搜尋效能可能下降。

**優化方向**：
- 確認 `車次`、`日期`、`分類`、`狀態` 欄位已建立索引，這些是最常用的篩選條件
- 「敏感物品」欄位建議獨立索引，因應後台查詢敏感物品列表的常用操作

## 優先級總覽

| 優先級 | 項目 | 理由 |
|---|---|---|
| 高 | 漸進式問答搜尋 | 直接提升尋回成功率，解決旅客體驗核心痛點 |
| 高 | 比對信心分數顯示 | 低成本、高信任度提升 |
| 高 | 敏感物品存取控制與紀錄 | 資安/個資風險，越晚改動成本越高 |
| 中 | 物品轉移紀錄結構化（ItemTransferLog） | 為未來流程分析鋪路，現階段改動成本較低 |
| 中 | AI 修正回饋記錄 | 長期累積資料以優化辨識準確率 |
| 中 | 配送規則可配置化 | 降低未來維護成本 |
| 低 | 多帳號權限分級 | 系統穩定後再導入，暫不影響核心功能 |
| 低 | 易腐壞食物倒數機制 | 邊緣情境，但具體驗加分效果 |

## 後續可深入規劃的項目

以下項目可在確認方向後，進一步產出具體的資料庫異動方案（ER 圖）或流程圖：

1. 漸進式問答搜尋的對話流程設計
2. `ItemTransferLog` 資料表結構設計與對應 API 異動規劃
3. 配送限制規則表的資料結構設計
4. 敏感物品分級權限的資料庫權限模型
